## 阶段 1: 基础设施

### 目的

**地基不稳则上层全部崩塌。** 目前代码虽然"能跑"，但有以下致命风险：
1. 没有 Git — 每次改代码都是裸奔，出问题无法回退
2. 认证凭空的 Better-Auth — 无 RBAC 角色权限，所有用户拥有相同权限
3. 操作日志靠手动调用 — 应该自动记录所有 API 操作
4. 仪表盘展示假数据 — 用户登录后看到的东西无意义
5. UI 组件无统一标准 — 按钮/颜色/圆角四处硬编码

### 任务清单

#### 任务 1.1 — Git 版本控制

| 项目 | 说明 |
|------|------|
| 为什么 | 防止代码丢失、方便回退、支持本地→服务器同步 |
| 做什么 | 安装 Git for Windows → 初始化仓库 → 首次提交 → 推送到 GitHub/Gitee 私仓 |
| 怎么验证 | `git log` 能看到提交历史 |

**执行步骤：**
1. 安装 [Git for Windows](https://git-scm.com/download/win)
2. 在项目根目录打开终端执行:
   ```bash
   git init
   git add .
   git commit -m "v0.1.0: 项目初始化，TikTok API 已打通"
   git remote add origin <你的私有仓库地址>
   git push -u origin main
   ```
3. 服务器端: `git clone` → 此后 `git pull` 即可更新

#### 任务 1.2 — RBAC 权限中间件

| 项目 | 说明 |
|------|------|
| 为什么 | 当前所有 API 没有权限控制。运营专员不应删改财务数据，财务不应改系统设置 |
| 做什么 | 后端 `middleware/` 目录下创建 RBAC 中间件；每个路由声明所需角色；JWT 中携带角色信息 |
| 怎么验证 | 用不同角色账号登录，无权访问的 API 返回 403 |

**角色定义（来自 PRD）：**

| 角色 | 权限范围 |
|------|----------|
| 超级管理员 (admin) | 所有功能 + 系统配置 + 用户管理 |
| 运营主管 (manager) | 看数据 + 管理订单/商品/达人/素材 |
| 财务 (finance) | 财务模块 + 只读其他 |
| 运营专员 (operator) | 订单处理 + 达人登记 + 素材上传 |

**中间件设计：**
```typescript
// middleware/rbac.ts
// 1. 从请求 Header 中提取 JWT Token
// 2. 解析出用户角色
// 3. 对照路由声明的所需角色列表
// 4. 不匹配则返回 403 Forbidden
```

#### 任务 1.3 — 审计日志中间件

| 项目 | 说明 |
|------|------|
| 为什么 | 谁在什么时候干了什么 — 商业软件的底线。当前需要手动调用 audit-log 记录，极度不可靠 |
| 做什么 | 创建全局中间件，自动记录每个 API 请求：用户ID、IP、路径、方法、请求体、响应状态码、耗时 |
| 怎么验证 | 任意操作后去审计日志页面查看，自动有记录 |

**关键设计：**
- 请求进来 → 记录开始时间
- 请求处理 → 拦截响应状态码
- 自动写入 `audit_log` 表
- 30 天自动清理旧日志

## 阶段 2: 核心链路

### 目的

**让系统有真实数据流入。** 阶段 1 补齐了地基，阶段 2 是把 TikTok 店铺的真实订单和商品"灌"进来。没有这一步，后面财务算利润根本没数据源。

开发顺序：**先订单后商品**。因为：
1. 订单同步是最高频需求（用户每天都要看卖了什么）
2. 商品同步 API 的返回结构和订单不同，需要独立处理
3. 订单进入后，用户自然要关联商品 → 驱动商品同步

### 任务清单

#### 任务 2.1 — 订单同步引擎

| 项目 | 说明 |
|------|------|
| 为什么 | 人工录订单不可能。TikTok Shop 每天几十上百单，必须自动拉取 |
| 做什么 | 创建 `services/tiktok-orders.ts`，周期性从 TikTok API 拉订单，去重入库 |

**技术方案：**

```
┌─────────────────────────────────────────────────┐
│  订单同步引擎                                     │
│                                                  │
│  触发方式:                                         │
│  ├─ 手动: 店铺管理页点击「同步订单」                    │
│  ├─ 定时: 每 10 分钟自动跑一次 (node-cron)             │
│  └─ Webhook: 预留 TikTok 回调通知                    │
│                                                  │
│  同步流程:                                         │
│  1. 查数据库这个店铺上次同步到哪了 (cursor)            │
│  2. 调 TikTok API: /order/202309/orders/search     │
│     ├─ 参数: shop_cipher, page_size=50              │
│     ├─ 参数: create_time_from (上次同步时间)          │
│     └─ 参数: sort_by=create_time, sort_type=ASC     │
│  3. 拿到订单列表 → 逐条入库 (INSERT OR IGNORE)       │
│  4. 对每条订单调 /order/202309/orders/{id} 拿详情    │
│     ├─ 订单明细 (order_items): SKU/Qty/Price        │
│     ├─ 支付信息: 交易金额/平台优惠/税费                 │
│     └─ 物流信息: 收货地址/物流状态                     │
│  5. 更新 sync_cursor 到最新同步时间                    │
│  6. 记录同步日志 (成功数/失败数)                       │
│                                                  │
│  去重策略:                                         │
│  order_id + shop_id 做唯一索引，重复自动跳过           │
└─────────────────────────────────────────────────┘
```

**TikTok 订单 API 关键节点：**

| API 端点 | 用途 | 关键返回字段 |
|----------|------|-------------|
| `POST /order/202309/orders/search` | 搜索订单列表 | order_id, order_status, create_time, update_time, payment_info |
| `GET /order/202309/orders/{order_id}` | 订单详情 | order_lines[](sku_id/sku_name/qty/price), payment(currency/amount/tax), delivery(address/tracking) |

**订单状态机（TikTok 标准）：**
```
UNPAID → ON_HOLD → AWAITING_SHIPMENT → PARTIALLY_SHIPPING
→ AWAITING_COLLECTION → IN_TRANSIT → DELIVERED → COMPLETED
CANCELLED / RETURN_REQUESTED / RETURN_IN_PROGRESS
```

#### 任务 2.3 — 商品同步引擎

| 项目 | 说明 |
|------|------|
| 为什么 | 手动录入商品 SKU 名称价格重量既慢又容易错。同步后自动获取 TikTok 上的完整商品信息 |
| 做什么 | 创建 `services/tiktok-products.ts`，从 TikTok API 拉商品列表和详情 |

**技术方案：**

```
┌─────────────────────────────────────────────────┐
│  商品同步引擎                                     │
│                                                  │
│  触发方式:                                         │
│  ├─ 手动: 商品管理页点击「从TikTok同步」              │
│  └─ 定时: 每 2 小时自动同步一次（商品变更频率低）      │
│                                                  │
│  同步流程:                                         │
│  1. 调 TikTok API: /product/202309/products/search │
│     ├─ 参数: shop_cipher, page_size=100             │
│     └─ 拿到商品 ID 列表                              │
│  2. 对每个 product_id 调详情 API:                    │
│     GET /product/202309/products/{product_id}       │
│     ├─ 基本信息: title/description/status/category  │
│     ├─ SKU 列表: skus[] → id/name/price/qty/weight │
│     ├─ 图片列表: main_images[] → url                │
│     └─ 物流信息: package_weight/dimensions          │
│  3. 入库: 商品主表 + SKU 子表 (INSERT OR UPDATE)     │
│     product.skus = JSON.stringify(skus) → 存 JSON  │
│  4. 记录同步日志                                     │
└─────────────────────────────────────────────────┘
```

**TikTok 商品 API 关键节点：**

| API 端点 | 用途 | 关键返回字段 |
|----------|------|-------------|
| `POST /product/202309/products/search` | 搜索商品列表 | product_id, product_status |
| `GET /product/202309/products/{id}` | 商品详情 | title, description, skus[], main_images[], category, package_weight |

**SKU 数据映射：**
```
TikTok SKU → 本地 product_sku 或 product JSON 字段
├─ sku_id       → skuId (平台 SKU ID)
├─ seller_sku   → sku (商家自定义编码)
├─ sku_name     → name (规格名称，如"红色-M码")
├─ price        → sellPrice (售价，原始币种)
├─ stock        → stock (库存)
├─ weight       → weight (重量，克)
└─ image        → image (SKU 图片)
```

### 目的

**这是 Bozone 最核心的差异化功能。** 大部分 ERP 只能看订单和流水，但 Bozone 要能算出"每条订单赚了多少、每个 SKU 利润率多少"。

### 任务清单

#### 任务 3.1 — OrderCost 费用明细表

| 项目 | 说明 |
|------|------|
| 为什么 | 这是 PRD 明确要求但当前缺失的核心表。没有它，利润计算只能靠公式硬算，无法追溯"这笔利润是怎么算出来的" |
| 做什么 | 建表 + 当订单进入时自动计算并写入费用明细 |

**表结构设计：**
```sql
CREATE TABLE order_cost (
  id TEXT PRIMARY KEY,
  order_id TEXT NOT NULL REFERENCES "order"(id),  -- 关联订单
  cost_item_id TEXT NOT NULL REFERENCES cost_item(id), -- 关联费用模板项
  name TEXT NOT NULL,        -- 费用名称（冗余，方便查询）
  charge_type TEXT NOT NULL, -- fixed / percentage / per_order / per_kg
  formula TEXT,              -- 计算说明，如 "¥42 × 2件 = ¥84"
  amount REAL NOT NULL,      -- 实际金额（RMB）
  currency TEXT DEFAULT 'CNY',
  created_at TEXT DEFAULT (datetime('now'))
);

CREATE INDEX idx_order_cost_order ON order_cost(order_id);
```

**自动计算时机：** 订单同步入库 → 触发费用计算 → 写入 order_cost 表

**计算逻辑（伪代码）：**
```
对每条订单:
  1. 加载该店铺的费用模板 (cost_item 表)
  2. 对每个费用项:
     if type == 'fixed':     amount = value                              # 固定 10 元
     if type == 'percentage': amount = 订单金额 × value%                 # 收入 5%
     if type == 'per_order': amount = value × 订单商品件数               # 每件 2 元
     if type == 'per_kg':    amount = value × Σ(商品重量 × 件数) / 1000 # 每公斤 15 元
  3. 写入 order_cost 表
```

#### 任务 3.2 — 利润计算引擎服务

| 项目 | 说明 |
|------|------|
| 为什么 | 利润计算不能分散在各处。集中到一个服务，所有模块调用同一个接口 |
| 做什么 | 创建 `services/profit-engine.ts`，提供订单利润/单品利润/店铺利润三个核心方法 |

**核心 API 设计：**
```typescript
// 利润引擎
class ProfitEngine {
  // 订单级利润：单条订单赚多少
  calculateOrderProfit(orderId: string): OrderProfit

  // 单品级利润：某个 SKU 在所有订单中的利润汇总
  calculateProductProfit(productId: string, dateRange?: DateRange): ProductProfit

  // 店铺级利润：某店铺某一时段的总利润
  calculateShopProfit(shopId: string, dateRange: DateRange): ShopProfit

  // 重新计算（费用模板修改后触发）
  recalculate(shopId: string): void
}

// 返回类型
interface OrderProfit {
  orderId: string
  revenue: number          // 收入 (已换算为 RMB)
  costs: CostBreakdown[]   // 各项费用明细
  totalCost: number        // 总费用
  netProfit: number        // 净利润
  profitMargin: number     // 利润率 %
}
```

**费用分摊规则（单品利润的关键）：**

```
一条订单包含 A商品×2 + B商品×1，总金额 ¥300

固定费用(包装¥10/单) → 按件数分摊:
  A: ¥10 × 2/3 = ¥6.67
  B: ¥10 × 1/3 = ¥3.33

比例费用(佣金5%) → 按金额分摊:
  A: ¥300 × 5% × (¥200/¥300) = ¥10
  B: ¥300 × 5% × (¥100/¥300) = ¥5

按重量费用(物流¥15/kg) → 按总重量分摊:
  A: 200g×2=400g, B: 300g×1=300g, 总700g
  A: ¥15 × (400/700) = ¥8.57
  B: ¥15 × (300/700) = ¥6.43
```

### C.3 数据映射：TikTok → 本地 product 表

```

skus[]               → skus (JSON.stringify 存储)
package_weight       → weight (克)
```

### C.4 SKU 存储策略

有两种方案：

**方案 A（推荐 — 简单直接）：**
`product.skus` 字段存储 JSON 字符串，前端解析展示。适合 SKU 数量少（<20）且不需要单独查询 SKU 的场景。

```json
[
  { "id": "SKU-RED-M", "sellerSku": "FRESH-M-RED", "name": "红色-M码",
    "price": 9.90, "stock": 150, "weight": 200, "image": "https://..." },
  { "id": "SKU-RED-L", "sellerSku": "FRESH-L-RED", "name": "红色-L码",
    "price": 11.90, "stock": 120, "weight": 250, "image": "https://..." }
]
```

**方案 B（未来扩展）：**
独立 `product_sku` 表，每个 SKU 一行。适合需要跨商品查 SKU、库存独立管理的场景。

**当前选择方案 A**，理由是 SKU 数量可控、查询足够、实现成本低。未来需要时再迁移到方案 B。

### C.5 商品图片处理

```
TikTok 返回 main_images 数组 → 取第一张存 mainImage 字段
完整图片列表存 images JSON 字段: ["url1", "url2", "url3"]
前端根据 images 数组展示轮播/缩略图
```

### C.6 成本配置（非 TikTok 数据，用户手动录入）

同步后的商品只有平台数据（售价/SKU/库存），以下字段需要用户手动填写：

| 字段 | 说明 | 用途 |
|------|------|------|
| `costPrice` | 采购成本（/件） | 利润计算的基础 |
| `weight` | 商品重量（克）| 物流费分摊 |
| `costBreakdown` | 其他成本（JSON）| 未来扩展 |

**UI 交互：** 商品列表页每行直接显示成本字段，点编辑按钮在侧滑面板中修改。
